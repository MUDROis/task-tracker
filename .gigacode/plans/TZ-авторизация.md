# Техническое задание: Система авторизации

## 1. Обзор

Текущая система авторизации реализована на стороне клиента и использует Firebase Realtime Database для хранения пользователей и задач. Система требует доработки для обеспечения безопасности и соответствия лучшим практикам.

## 2. Текущая реализация

### 2.1 Где реализована

**Файлы:**
- `app.js` (строки ~150-200) — основная логика авторизации
- `index.html` (строки ~12-30) — форма входа
- `firebase-config.js` — конфигурация Firebase
- `style.css` (строки ~60-100) — стили формы входа

### 2.2 Процесс авторизации

1. **Отображение формы входа** (`showLoginPage()`)
   - Если сессия отсутствует, показывается страница входа
   - Форма с полями "Логин" и "Пароль"

2. **Проверка учетных данных**
   - Получение пользователя из Firebase по логину
   - Хеширование введённого пароля функцией `hashPassword()`
   - Сравнение хеша с хешем в базе

3. **Создание сессии**
   - Если проверка успешна, создаётся объект `currentUser = { login, role }`
   - Данные сохраняются в `localStorage` под ключом `'taskTracker_session'`

4. **Отображение главной страницы** (`showMainPage()`)
   - Скрывается форма входа
   - Показывается основное приложение
   - Отображается роль пользователя (Руководитель/Сотрудник)
   - Кнопка "Управление" показывается только для руководителя

5. **Восстановление сессии**
   - При загрузке страницы проверяется `localStorage`
   - Если сессия найдена — автоматический вход
   - Если нет — показ формы входа

### 2.3 Функции хеширования

```javascript
function hashPassword(password) {
    let hash = 0;
    for (let i = 0; i < password.length; i++) {
        const char = password.charCodeAt(i);
        hash = ((hash << 5) - hash) + char;
        hash = hash & hash;
    }
    return hash.toString(16);
}
```

**Проблема:** Это некриптографический хеш (djb2), не подходит для хранения паролей.

### 2.4 Хранение данных в Firebase

**Структура:**
```
teams/
  team_main/
    users/
      admin/
        login: "admin"
        passwordHash: "5d41402abc4b2a76b9719d911017c592" (MD5 для "1")
        role: "admin"
        color: "#3b82f6"
        email: ""
    tasks/
      ...
```

**Правила безопасности (Firebase):**
```javascript
{ "rules": { ".read": true, ".write": true } }
```
**Проблема:** Открытый доступ для всех — любой может читать и писать данные.

## 3. Проблемы безопасности

| Проблема | Описание | Уровень риска |
|----------|----------|---------------|
| **Хранение паролей** | Используется простой хеш без соли и итераций | Критический |
| **Открытая база данных** | Firebase имеет правила `.read: true, .write: true` | Критический |
| **Нет защиты от перебора** | Нет блокировки после неудачных попыток | Средний |
| **Нет HTTPS** | Если приложение не развернуто по HTTPS, данные передаются в открытом виде | Высокий |
| **Сессия в localStorage** | localStorage может быть украден через XSS | Средний |
| **Нет валидации** | Нет проверки сложности паролей при создании | Средний |

## 4. Требования к доработанной системе

### 4.1 Обязательные требования

#### 4.1.1 Хранение паролей
- Использовать криптографически стойкий алгоритм: **bcrypt** или **scrypt**
- Обязательно добавить соль (salt) для каждого пароля
- Минимум 10 000+ итераций хеширования

#### 4.1.2 Безопасность Firebase
- Изменить правила Firebase на режим "Только авторизованные пользователи"
- Использовать Firebase Authentication вместо кастомной системы
- Реализовать правила безопасности на уровне базы данных:

```javascript
{
  "rules": {
    "teams": {
      "$teamId": {
        "users": {
          "$userId": {
            ".read": "auth != null && (auth.uid === $userId || root.child('teams/' + $teamId + '/users/' + auth.uid + '/role').val() === 'admin')",
            ".write": "auth != null && (auth.uid === $userId || root.child('teams/' + $teamId + '/users/' + auth.uid + '/role').val() === 'admin')"
          }
        },
        "tasks": {
          ".read": "auth != null",
          ".write": "auth != null"
        }
      }
    }
  }
}
```

#### 4.1.3 Защита от атак
- Ввести блокировку после 5 неудачных попыток входа (на 15 минут)
- Логирование попыток входа (для администратора)
- Валидация всех входных данных (防 XSS)

#### 4.1.4 Управление сессиями
- Использовать `HttpOnly` cookies вместо localStorage (если возможно)
- Реализовать таймаут сессии (например, 30 минут бездействия)
- Кнопка "Выйти со всех устройств"

### 4.2 Желательные требования

#### 4.2.1 Firebase Authentication
- Миграция на Firebase Auth с поддержкой:
  - Email/Password
  - Google, GitHub, Microsoft (опционально)
- Управление паролями через Firebase (сброс пароля, подтверждение email)

#### 4.2.2 Двухфакторная аутентификация (2FA)
- Поддержка Google Authenticator или SMS
- Обязательная 2FA для ролей руководителя

#### 4.2.3 Аудит и логирование
- Логирование всех операций входа/выхода
- Логирование изменений паролей
- Возможность просмотра активных сессий

#### 4.2.4 Управление профилем
- Изменение пароля пользователем
- Сброс пароля по email
- Удаление аккаунта
